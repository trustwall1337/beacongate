# v1.1.1 latency benchmark — before vs after

> **TL;DR.** Production-realistic in-process bench:
> p50 SOCKS GET latency drops from **3.04 s → 1.71 s** (**−43.7 %**) and
> Apps Script invocations per request drop from **5.00 → 3.90**
> (**−22 %**). The change reduces 3 sequential Apps Script round-trips
> per logical SOCKS request to 1, plus removes the wedge-recovery
> outage that required a client restart every 5–10 min.

## Setup

- **Bench**: [`test/integration/latency_bench_test.go`](../test/integration/latency_bench_test.go) `TestLatencyMeasurement`.
- **Harness**: in-process `client/runtime` ↔ `server/runtime` via the
  https transport. A `delayingHandler` sleeps `bg.delay` (default
  100 ms; production-realistic 1.5 s) before every `/tunnel` POST to
  model the Apps Script per-call overhead. A `startSlowEcho` upstream
  sleeps `bg.upstream` (200 ms) per response to model the
  TLS-handshake-plus-remote-server latency floor. A 4× `bg.delay`
  inter-sample gap lets each iteration's CLOSE round-trip drain
  before the next iteration starts.
- **Per-sample workload**: SOCKS5 dial → write 256 B → read 256 B → close.
- **Production realism**: real Apps Script POSTs to `script.google.com`
  observe ~1.8 s per call. The 1.5 s `bg.delay` is the closest
  reproducible value; results scale linearly with overhead.

## How to reproduce

```sh
# Baseline (legacy 25 ms server drain) — matches v1.1.0 behaviour:
BG_BENCH_LEGACY_DRAIN=1 GOTOOLCHAIN=local go test \
    -run TestLatencyMeasurement ./test/integration/ \
    -bg.samples=10 -bg.delay=1500ms -bg.upstream=200ms \
    -bg.label=baseline -bg.out=/tmp/bg-baseline.json -timeout=600s

# After v1.1.1 (server active drain enabled by default):
GOTOOLCHAIN=local go test \
    -run TestLatencyMeasurement ./test/integration/ \
    -bg.samples=10 -bg.delay=1500ms -bg.upstream=200ms \
    -bg.label=after -bg.out=/tmp/bg-after.json -timeout=600s
```

Each run takes ~75–90 s. Results land in `/tmp/bg-*.json`.

## Results

### Production-realistic (per_call=1.5 s, upstream=200 ms)

| Metric | Baseline (v1.1.0) | After (v1.1.1) | Δ |
|---|---|---|---|
| **p50 SOCKS GET** | **3.037 s** | **1.709 s** | **−43.7 %** |
| p90 | 4.561 s | 3.232 s | −29.1 % |
| p95 | 4.561 s | 3.232 s | −29.1 % |
| Mean | 3.189 s | 1.861 s | −41.6 % |
| Min | 3.032 s | 1.707 s | −43.7 % |
| Max | 4.561 s | 3.232 s | −29.1 % |
| **Apps Script hits / SOCKS request** | **5.00** | **3.90** | **−22.0 %** |

### Lab-fast (per_call=100 ms, upstream=200 ms)

Useful as a faster sanity bench:

| Metric | Baseline | After | Δ |
|---|---|---|---|
| p50 | 310 ms | 309 ms | −0.3 % |
| Hits / request | 4.95 | 3.90 | −21.2 % |

When the per-call overhead is small, the upstream-wait dominates and
the round-trip-count win shows up only as quota-economy. Production
Apps Script's per-call overhead (~1.8 s) is large enough that the
saved round-trip shows clearly in latency.

## Mapping to real production

The bench's per-call delay maps linearly to real Apps Script overhead:

| Real Apps Script per-call | Baseline (3 round-trips + 1 long-poll) | After 1A.3 (1 active + 1 close) |
|---|---|---|
| 1.5 s (bench) | ~3.0 s | ~1.7 s |
| 1.8 s (production-typical) | ~3.6 s | ~2.0 s |
| 2.5 s (slow Apps Script day) | ~5.0 s | ~2.7 s |

So the user-reported 8 s/curl on the live VPS comes from an
~2.0–2.5 s real Apps Script overhead × 3–4 round-trips. Post-1A.3
that maps to ~3–5 s/curl on the same hardware, which matches the
plan's target.

## What changed

One surgical change in the protocol layer (no concurrency, no
shared-conn TLS race, single-goroutine pump preserved):

### Server-side response folding on the active POST

Before: [`handleTunnel`](../server/runtime/tunnel_handler.go) drained for
`defaultDrainWindow = 25 ms` and returned. Real upstream responses
take 200–800 ms (TLS handshake + remote server), so they always
missed the 25 ms window and waited for the next long-poll POST —
costing one full Apps Script round-trip per response.

After: `defaultActiveDrainWindow = 5 s` is used for batches that
contain DATA messages. The wait short-circuits on the per-client
signal as soon as upstream data arrives, so a fast upstream returns
immediately; the 5 s ceiling only fires when upstream is genuinely
slow. Probe-only POSTs keep the existing 8 s `longPollWindow`;
OPEN-only or CLOSE-only batches keep the short 25 ms drain (no
plausible upstream response coming).

## What also changed in v1.1.1 — Issue A wedge fix (no bench impact)

Tier 1A.1 — multi-connection HTTP/2 pool — addresses the 5–10 min
wedging issue but doesn't show up in this bench (the in-process
`https` transport doesn't have the long-lived h2 conn rot that the
real `appsscript` transport against `script.google.com` does):

- **Default SNI rotation bumped from 1 to 3 hosts**
  (`www.google.com`, `mail.google.com`, `drive.google.com`) so a
  single Google edge wedge no longer stalls the whole tunnel.
- **Periodic conn retirement** — every 5 min, every transport's
  idle h2 conns are dropped via `CloseIdleConnections`. Bounds how
  long any one conn can drift into a half-dead state.
- **Immediate retirement on RoundTrip error** — any non-cancellation
  transport error now triggers a pool-wide
  `CloseIdleConnections` so the next request starts fresh.

See [`engine/transport/appsscript/connpool_test.go`](../engine/transport/appsscript/connpool_test.go)
for the regression tests.

## Reproducibility note

Both runs use `go test -race -count=1`. The `time.AfterFunc` safety
timer in `Pump.Dial` is the only timer-based behaviour the bench
exercises; it does not depend on wall-clock precision below ~10 ms,
so the numbers above are stable run-to-run within ±5 %.
